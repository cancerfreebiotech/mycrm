本章では myCRM の管理者向け機能を説明します。ほとんどのページはサイドバーの「管理」エリアにあり、パスは `/admin/` で始まります。一部の機能は一般メンバーに委任できますが、それ以外は Super Admin 専用です。

## 権限とアクセスモデル

管理権限は 2 階層です。

- **Super Admin**：すべての管理機能にアクセスでき、ロール・権限委任・モジュールの切り替え・システム運用を担当します。
- **一般メンバー（member）**：既定では管理項目が表示されません。Super Admin が「ユーザー管理」で個別メンバーに機能を項目単位で委任でき、委任されていない項目はそのメンバーのサイドバーに表示されません。

**一般メンバーに委任できる機能**：タグ、未割り当てメモ、メールテンプレート、プロンプト、国、ニュースレター、認識失敗レビュー、重複連絡先、名刺（CamCard）取り込み、ゴミ箱、連絡先エクスポート、一斉メール、「ユーザー管理」（委任されたメンバーは MFA リセットと Telegram ID 編集のみ可能）。

**全メンバーに開放**：レポート（各自が自分のスケジュールを管理。Super Admin は全員分を閲覧可能）。

**Super Admin 専用**：AI モデル、システムヘルス、MCP トークン、MCP アクティビティ、Email 復活、共有 Email、フィードバック、配信停止と送信可否、データ主体リクエスト（DSAR）、組織設定、ニュースレター概要、監査ログ。

> フロントエンドの PermissionGate は入口を隠すだけです。すべての `/api/admin/*` ルートはサーバー側で権限を**再検証**します（未ログインは 401、権限不足は 403）。

## ユーザー管理

パス：`/admin/users`。アクセス権：Super Admin、または「ユーザー管理」を委任されたメンバー（後者は MFA リセットと Telegram ID 編集のみ）。

**概要**：ログイン実績のある全メンバーのアカウント、ロール、機能委任、アカウント状態を管理します。列（すべて並べ替え可能）：氏名、Email、Telegram、Teams 連携、ロール、最終ログイン、MFA 状態。モバイルではカード表示。

**使い方**：

- **ロール管理**（Super Admin のみ）：「Super Admin に昇格」／「一般メンバーに降格」で切り替え。自分のロールは変更できず、Super Admin は常に最低 1 名を維持します。
- **停止 / 離職**（Super Admin のみ）：「停止」でアカウントを無効化（二段階確認）、「再有効化」で復帰。停止は**即時反映**され、対象者は次の操作でログアウトされログイン画面に戻ります（セッション期限を待ちません）。
- **機能委任**（Super Admin のみ、member に表示）：メンバーごとに権限チップが並び、クリックで付与／解除。付与済みは ✓ 表示。
- **MFA リセット**：認証器を設定済みのメンバーに「リセット」が表示され、確認後にすべての MFA 要素を削除、次回ログイン時に再設定が必須になります。
- **Telegram ID 編集**：Telegram 列でインライン編集。正の整数のみで、9 桁未満は警告、空欄で保存すると連携解除。
- **メンテナンスモード**（Super Admin のみ）：ページ上部のトグル。ON にすると一般ユーザーはメンテナンスページへ誘導され、Super Admin のみ継続利用できます。

> Super Admin（`pohan.chen@cancerfree.io`）は停止・降格できません。アカウントは初回ログイン時に自動作成され、`@cancerfree.io` ドメインの Microsoft アカウントのみログインできます。

## 組織設定とブランディング

パス：`/admin/org-settings`（Super Admin のみ）。

**概要**：コードの変更や再デプロイなしに、全社共通の組織情報・ニュースレターのブランディング・メール関連パラメータを調整します。設定には **60 秒のキャッシュ**があり、保存後 1 分以内に反映されます。空欄はシステム既定値を使用します。

**設定できる項目**：組織名、ログインを許可する Email ドメイン（バックエンドのログインドメイン検証元、カンマ区切り）、ニュースレターのロゴ URL、ニュースレターの返信先 Email、会社サイト / Facebook / LinkedIn（フッターのソーシャルリンク）、システム通知の受信者。

**v7.9.4** 以降、次の 7 つのメール／システムパラメータもここで設定します（以前はコードに直書き）：送信者の表示名、社内メールドメイン、組織メールドメイン、BCC アーカイブ用ドメイン、物理的な郵送先住所（ニュースレターフッター、CAN-SPAM 対応）、組織オーナー Email（アラートやシステムメールの受信者）、アプリケーション URL（メールのリンクで使う外部 URL）。

**モジュールスイッチ**：機能モジュール全体をワンクリックで停止でき、同じく 60 秒以内に反映されます。

- **Hunter.io 自動補完**：OFF にすると新規連絡先の会社名・役職の自動補完が停止し、システムヘルスでの補完実行時に「モジュール停止中」と表示されます。
- **AI アシスタント**：OFF にすると Web 版と Telegram `/ai` の AI アシスタントが停止します。

> すべての変更は監査ログに記録されます（アクション：組織設定の変更）。

## タグ管理

パス：`/admin/tags`。アクセス権：Super Admin、または「タグ」を委任されたメンバー。

**概要**：連絡先を分類するタグを管理します。一覧にはタグ名、使用中の連絡先数、作成日を表示します。

**使い方**：

- **追加**：名前を入力して「追加」。名前は重複不可。
- **名前変更**：✏️ でインライン編集し、Enter か ✓ で保存。
- **Email ブラックリスト**：盾アイコンで切り替え。ブラックリストにしたタグの連絡先は、**送信およびニュースレターのリスト作成から自動的に除外**され、タグ名の横に赤い「ブラックリスト」マークが表示されます。
- **削除**：🗑 で確認後に削除。削除するとすべての連絡先とそのタグの関連付けも解除されます。

## 国管理

パス：`/admin/countries`。アクセス権：Super Admin、または「国管理」を委任されたメンバー。

**概要**：連絡先で選択できる国リストを管理します。列：ISO コード、国旗絵文字、中/英/日の名称、状態。コードや名称で並べ替え可能。

**使い方**：

- **追加**：「国を追加」をクリック。コードは 2 文字の ISO 3166-1（自動で大文字化）。一般的なコード（TW、JP、US など）を入力すると 3 言語の名称と国旗が**自動入力**されます。中国語名と英語名は必須。
- **編集**：鉛筆で名称と国旗を変更。コードは作成後は変更不可。
- **有効 / 無効**：状態バッジで切り替え。無効にした国はメニューに出ませんがデータは保持されます。
- **削除**：ゴミ箱アイコンからインライン確認で削除。

## AI プロンプト

パス：`/admin/prompts`。アクセス権：Super Admin、または「プロンプト」を委任されたメンバー。

**概要**：各 AI 機能が使うシステムプロンプトの**組織レベルの上書き**を設定します。空欄なら組み込みの既定値を使用します。設定対象：名刺認識、タスク解析、メール生成（「ユーザー編集可能」と表示）、ドキュメント生成、会議解析。

**使い方**：該当のテキスト欄にカスタムプロンプトを入力して「保存」。**空欄のまま保存**すると上書きを削除し、既定値に戻します。既定値を使っている項目では、組み込みプロンプトが下部に参考表示されます。

## AI モデルとエンドポイント

パス：`/admin/models`（Super Admin のみ）。

**概要**：AI モデルは 2 階層構造です。**AI プロバイダー（Endpoint）** の下に複数の **AI モデル（Model）** がぶら下がります。どのモデルを使うかは下部の「機能の割り当て」で AI 機能ごとに決まります。

**プロバイダーの管理**：

- **Endpoint を追加**：プロバイダー名、**タイプ**、Base URL、API Key（省略可、後述）を入力。
- **タイプ**（v8.0.0）：`Google (Gemini)` は公式 Gemini API で、AI アシスタントの function calling と Social Briefing の Google 検索 grounding に対応します。`OpenAI 互換` は `/chat/completions` を提供する任意のサービス——Portkey、OpenRouter、または**オンプレミス**の Ollama／vLLM／LM Studio に対応します。タイプは一覧のドロップダウンから直接切り替えられます。
- **API Key は省略可**（v8.0.0）：オンプレミスやキー不要のエンドポイントは空欄にできます。**オンプレミス利用時の注意**：Base URL はデプロイ環境（Vercel）から到達できる外部アドレスである必要があります——社内 IP は不可のため、Cloudflare Tunnel や Tailscale Funnel のような経路を使ってください。「テスト」ボタンで経路全体を確認できます。
- **API Key の変更／名前の変更**：キーは既定でマスクされ、平文表示に切り替え可能。プロバイダー名は鉛筆✏️アイコンからインラインで編集できます（v8.0.1）。Base URL は作成後は変更不可。
- **有効 / 無効 / 削除**：状態トグルで操作。配下のモデルを先に消さなくてもプロバイダーを直接削除できます。

**モデルの管理**：先にプロバイダーを選択すると、その配下のモデル一覧が表示されます。

- **Model を追加**：Model ID（API 用、例 `gemini-2.5-flash`）と表示名を入力。
- **無効化**：機能の割り当てのドロップダウンから消え、このモデルを割り当てていた機能はシステム既定に切り替わります。
- **削除**：🗑 で確認後に削除。

**テストボタン**（v8.0.0）：Endpoint 行と Model 行それぞれに「テスト」ボタンがあります。押すとサーバー側で極小の AI 呼び出しを 1 回実行し、結果はボタンの横にすぐ表示されます（緑の ✅ は成功＋所要時間、赤の ❌ は失敗理由）。「前回テスト」の時刻と結果も記録され、再読み込み後も表示されます。Endpoint に有効なモデルがない場合は、単純な疎通確認（HTTP の応答があれば合格）にフォールバックします。

### 機能の割り当て

ページ下部で、8 つの AI 機能をそれぞれ特定のモデルに割り当てられます。未割り当ての場合は「システム既定」（環境に組み込まれた経路で、エンドポイント一覧には含まれません）になります。各行に「現在有効」が表示され、実際に使用中のモデルと経路がわかります。

**8 つの機能**：AI アシスタント（Web ＋ Telegram `/ai`）、Social Briefing、やり取り記録の整形、日次フィードバックの一次判定、AI レビュー（重複候補の提案＋統合レビュー＋メモのマッチング）、ニュースレターの文章校正、ニュースレターの翻訳、名刺認識の既定（名刺認識／Bot コマンド解析／AI メール生成で常に使われる組織既定。未割り当てならシステム既定にフォールバック——個人の選択レイヤーはもうありません）。

どの機能もどのエンドポイントのモデルにも割り当てられます。AI アシスタントと Social Briefing は Google 専用の機能（function calling／Google 検索 grounding）が必要なため、OpenAI 互換エンドポイントに割り当てると黄色い警告「動作しない、または品質が低下する可能性があります」が表示されます——AI アシスタントは OpenAI 形式のツール呼び出しで実行するようになり（エンドポイントが tools に対応している必要があります）、Social Briefing はリアルタイムの検索ソースがなくなります。各行にもテストボタンがあり、割り当て済みならそのモデルをテストして記録し、未割り当てならシステム既定の経路をテストして結果を表示のみ（記録しない）します。割り当ての変更は**即時反映**されます（最大で約 1 分の遅延）。

## メールテンプレート

パス：`/admin/templates`。アクセス権：Super Admin、または「メールテンプレート」を委任されたメンバー。

**概要**：送信時に適用できるテンプレート（件名、本文、添付）を管理します。一覧には名前、件名、作成日、添付数を表示。行を展開すると本文をプレビュー（HTML は DOMPurify でサニタイズ）。

**使い方**：「テンプレートを追加」または鉛筆で編集画面を開きます。

- 名前（必須）と件名。
- 本文は WYSIWYG（TipTap）エディタ。
- **AI 生成**：説明を入力すると AI が本文を生成してエディタに挿入（既存本文は参考として利用）。
- **添付**：ファイルをアップロード（1 ファイル最大 5MB）。既存テンプレートの編集時は即保存、新規テンプレートは保存時にまとめて書き込み。添付は送信メールに付けてダウンロードできます。

テンプレートの削除はゴミ箱からインライン確認。添付を削除すると Storage からも同時に削除されます。

## レポート管理

パス：`/admin/reports`。アクセス権：**全メンバー**がページを開いてレポートを生成し、自分のスケジュールを管理できます。Super Admin はさらに「作成者」列と全員のスケジュールを閲覧できます。

**概要 / 使い方**：

- **今すぐ生成**：期間を選択 → フィルター（タグ、やり取りの種類、作成者、国）を適用 → 「生成」でテーブルプレビュー、または「Excel ダウンロード」。
- **定期スケジュール**：「スケジュールを追加」で名前、頻度（毎週／毎月／カスタム Cron）、対象日数、宛先（カンマ区切り）を設定。各スケジュールは有効／無効・編集・削除でき、「前回実行」に直近の送信時刻と結果を表示。
- **送信方法**：バックエンドの自動処理が担当し、1 時間ごとにチェックして期限到来分を自動送信します。手動トリガーも外部メールアカウントの連携も不要です。

## ゴミ箱

パス：`/admin/trash`。**実質的に Super Admin 専用**（PermissionGate に加えて super_admin チェックがあります）。

**概要**：連絡先は削除するとまずゴミ箱へ移動し、復元または完全削除できます。列：チェックボックス、氏名、会社、削除者、削除日時。上部のバナーに現在の件数を表示。

**使い方**：

- **復元**：正式な一覧へ戻す（確認あり）。
- **完全削除**：1 件を完全に削除（確認あり、復元不可）。
- **選択削除 / 全件削除**：一括、またはゴミ箱を一度に空に。
- 氏名をクリックすると詳細ダイアログで全項目・タグ・名刺画像・直近 10 件のやり取りを確認でき、その中で復元や完全削除も可能。名刺画像は非公開バケットにあり、短時間有効の署名付きリンクで表示されます。

## 重複連絡先レビュー

パス：`/admin/duplicates`。アクセス権：Super Admin、または「重複連絡先」を委任されたメンバー。

**概要**：重複の疑いがある連絡先ペアを見つけ、1 組ずつ統合または無視します。「スキャン」でペアを再検出。ペアは **Email 完全一致**（赤）と **氏名類似**（黄、類似度％付き）の 2 種に分かれます。

**使い方**：各ペアは 2 枚のカードを並べて表示し、次を実行できます。

- **左を残す／右を残す**：残す側を主として統合確認を開く（緑＝残す、赤＝削除される。残す側の項目が優先され、相手の関連データはすべて統合。統合は復元不可）。
- **重複ではない**：このペアを無視し、以後表示しない。
- **AI 判定**（v7.6.0）：同一人物かを AI に分析させ、判定（同一／別人／不確実）、確信度、理由を返します。「同一」の場合は「AI の統合提案を採用」が表示。AI は参考で、最終確認は人間が行います。
- **一括処理**：各アクションをバッチに「追加」でき、上部に件数を表示。「実行」で順に処理（無視を先に、統合は 1 件ずつ呼び出し）。

## 名刺（CamCard）取り込みレビュー

パス：`/admin/camcard`。アクセス権：Super Admin、または「名刺取り込み」を委任されたメンバー。

**概要**：一括取り込みした名刺は認識後にまずレビュー待ちリストに入り、人による確認後に正式な連絡先になります。リストは**会社**ごとにグループ化され、1 ページ 20 件。

**フィルター**：検索（自動クエリ）、国、レビュー担当（または未割り当て）、重複あり、Email あり、並べ替え（新しい順／古い順）。

**1 枚ずつレビュー**：各カードは表裏のサムネイルと認識項目を表示し、4 つの操作があります。

- **追加**：現在の重要度／言語／タグで新規連絡先を作成。Email 完全一致の重複と判定された場合はロックされ、統合かスキップを使います。
- **編集**：認識項目を修正して保存。
- **統合**：既存連絡先に統合（検索可、重複検出時は自動選択。「空欄のみ補完」と「上書き」モード）。
- **スキップ**：作成せず飛ばす。

各カードで**重要度**（H/M/L）、**言語**（中/EN/日）、**タグ**を設定でき、追加時または一括確認時に書き込まれます。

**一括操作**：グループ一括追加（そのグループの重複なしカードをワンクリック確認）、チェックボックス一括確認（進捗バー付き）、レビュー担当の割り当て（担当タグを入力）、やり取り日時の一括設定。「追加」では**バックデート**（既定 `2000-01-01`）を指定でき、大量の古い名刺が「最近追加」を埋めないようにします。

## 認識失敗レビュー

パス：`/admin/failed-scans`。アクセス権：Super Admin、または「認識失敗レビュー」を委任されたメンバー。

**概要**：Bot で撮影・アップロードしたが AI が氏名を認識できなかった名刺がここに入り、見落としを防ぎます。既定では未レビュー項目のみ表示。「レビュー済みを表示」で処理済みの記録も確認できます。

**使い方**：名刺画像（非公開バケットの署名付きリンク）を確認 →「→ 連絡先を作成」で画像と取得元を引き継いで新規連絡先ページへ移動し手入力で保存 → 本ページに戻り「完了にする」でリストから外します。2 つの手順は独立しており、連絡先を作成した後も完了にする必要があります。完了にするとレビュー担当の email と時刻が記録されます。

## 共有 Email 連絡先

パス：`/admin/shared-emails`。アクセス権：Super Admin、または「一斉メール」を委任されたメンバー。

**概要**：2 人以上の連絡先が同じ email を共有しているケース（夫婦、家族、共有アドレス）を洗い出し、一斉送信前の参考にして同じアドレスへの重複送信を防ぎます。上部の統計に共有グループ数と関係する連絡先数を表示し、email／氏名／会社でリアルタイム絞り込みができます。

**内容**：1 行に共有 email が 1 件で、そのアドレス、使用人数、該当連絡先（連絡先ページへのリンク）を表示。読み取り専用で、統合はここでは行いません。統合は「重複連絡先レビュー」を使います。

## 配信停止と送信可否

パス：`/admin/suppressions`（Super Admin のみ）。

**概要**：任意の email について「**送っていいか、なぜか**」を調べ、全システムの抑制リストを俯瞰します。判定は**厳格な和集合**で、いずれか 1 つの来源で抑制されれば送信不可と判定します。2 つのビュー：**単一 email 照会**（判定カード＋来源別の状態表）と、**入力しない既定ビュー**（来源別の集計＋直近 50 件の抑制記録）。本ページは**読み取り専用**で、抑制の追加や解除は行いません。

### 5 つの抑制来源

| 来源 | 判定条件 |
|------|----------|
| 連絡先の配信停止（opt-out） | `contacts.email_opt_out = true`（CRM 直接送信で尊重） |
| 連絡先の Email 状態 | `contacts.email_status` が空でない（送信 worker は空でない値すべてを抑制：bounced／invalid／unsubscribed／recipient_blocked／spam_report） |
| ブラックリスト | `newsletter_blacklist` に存在（多くは SendGrid のハードバウンス／無効アドレス／ブロック由来） |
| グローバル配信停止 | `newsletter_unsubscribes` に存在（SendGrid の配信停止とスパム報告を含む） |
| 購読者の配信停止 | `newsletter_subscribers.unsubscribed_at` が空でない |

先頭 2 つは連絡先の `email` または第 2 email と照合し、未削除の連絡先のみを対象とします。残り 3 つは email を完全一致（大文字小文字を区別しない）で照合します。

### 毎日の SendGrid 抑制取り込みとの関係

これらの来源の多くは、スケジュールジョブ `POST/GET /api/sendgrid/import-suppressions` により毎日埋め戻されます（cron `0 19 * * *`、台北時間で翌日 03:00 頃）。SendGrid の直近 90 日のハードバウンス／無効アドレス／配信停止／ブロック／スパム報告をページングで取得し、`contacts.email_status` に書き戻します。CRM にないアドレスはブラックリストへ、配信停止／スパム報告はグローバル配信停止へも書き込み、一致した連絡先には system のやり取り記録を追加します。Super Admin はニュースレターのリストページから手動実行もできます。要するに **このページはビューアにすぎず**、実際にデータを流し込むのはこの毎日の cron です。

## Email 復活

パス：`/admin/email-recovery`。アクセス権：Super Admin、または「重複連絡先」を委任されたメンバー。

**概要**：email がバウンス／無効になった連絡先について、同じ人物が転職後に残した新しい名刺（新 email 付き）を見つけ、古い連絡先の email をワンクリックで置き換えます。上部の統計にバウンス／無効の総数と「候補の新 email あり」の件数を表示し、「候補あり」と「すべて」を切り替えられます。

**使い方**：各項目は壊れた email、状態、バウンス理由をアラート色で示します。候補が見つかると、相手の氏名・会社・新 email が緑色で表示されます。

- 「この email で置き換え」で候補の email を古い連絡先に取り込み（候補連絡先の統合も可能）、または「手入力」欄に入力して置き換えます。
- 置き換え前に確認が入ります（連絡先データを変更するため）。完了した項目は薄くなり「置換済み」と表示。候補は氏名／会社で照合されるため、同一人物であることを人間が確認してから置き換えてください。

## データ主体リクエスト（DSAR）

パス：`/admin/dsar`（Super Admin のみ）。

**概要**：email を入力すると、システム内の関連連絡先をすべて見つけ、それぞれの**個人データフットプリント**（やり取り記録、名刺、ニュースレター受信数）を一覧します。GDPR のデータ主体からの開示／エクスポート／削除リクエストへの対応に使います。

- 連絡先の `email` または第 2 email を **完全一致（大文字小文字を区別しない）** で照合。
- **削除済み（ゴミ箱）の連絡先も含み**、「削除済み」と表示します。データ主体リクエストはすべての記録を対象にする必要があるためです。
- そのアドレスの**送信可否判定**も表示します（判定ロジックは「配信停止と送信可否」と共通）。

**使い方**：完全な email を入力 →「照会」。上部に送信可否カード、下部に一致する全連絡先とフットプリントが並びます。各行で**表示**（`/contacts/{id}` を開く）または個人データ JSON の**エクスポート**ができます。

**個人データエクスポート**：`/api/contacts/{id}/export` を呼び出し、単一の JSON（`contact-{id}-export.json`）を返します。連絡先本体、名刺、写真、やり取り、タスク、ニュースレター受信、Email イベントを集約します。画像本体は Storage に残るため、エクスポートには URL／パスのみが含まれます。

> 説明責任のため、**照会のたびに監査ログへ記録**されます（アクション：DSAR 照会。対象は照会した email）。エクスポートも監査ログに記録されます（アクション：GDPR データエクスポート）。本ページは読み取り専用で、データの削除・変更は行いません。

## システムヘルス

パス：`/admin/health`（Super Admin のみ）。

**概要**：システムの稼働状態を 1 ページで俯瞰します。上部で「今すぐチェック」、または「30 秒ごとに自動更新」で継続監視。各ブロックはそれぞれのAPI から取得するため、あるブロックが失敗してもそのブロックのみエラーを表示し、単独で再試行できます。

- **外部サービス状態**：Supabase、Gemini、Portkey（AI ゲートウェイ）、Telegram Bot、SendGrid、Teams Bot。状態バッジ（正常／異常／未設定）＋レイテンシバー（緑 <500ms、黄 <2000ms、赤 それ以上）。
- **Hunter.io（Email 検索）**：API Key を貼り付けて保存。統計は email なし総数／未検索／検索したが見つからず／今月検索済みと残り credits を表示。「検索開始」で未検索の連絡先の email を照会、「検索状態をリセット」でフラグを解除（確認あり）。組織設定で「Hunter.io 自動補完」が OFF の場合は「モジュール停止中」と表示し、credits を消費しません。
- **スケジュール心拍**（v7.4.0）：各バックグラウンドスケジュール（cron）の直近実行状況——正常／遅延／エラー／記録なし。前回完了時刻と所要時間を表示。
- **デッドレター**（v7.4.0）：各テーブルの処理失敗件数。`pending_contacts` と `contact_briefings` は「再キュー」でき、`failed_scans` は「認識失敗レビュー」へのリンクを提供。
- **今月の使用量と予算しきい値**（v7.4.0、しきい値編集は v7.9.5）：AI 呼び出し回数、AI 入出力トークン、送信メール数、ニュースレター送信数を前月と比較。各カードに「月間上限」を入力可能——空欄＝無制限。設定するとプログレスバーと割合を表示（緑 <80%、黄 80–99%、赤 ≥100%）、80%／100% 到達時にシステムが組織オーナーへ自動通知します。

## 監査ログ

パス：`/admin/audit-log`（Super Admin のみ、v7.4.0 で追加）。

**概要**：機微／特権操作の監査証跡を確認します。データは `admin_actions` テーブル由来で、1 ページ 20 件、ページ送り・絞り込み・CSV エクスポートに対応。列：時刻（台北タイムゾーン）、操作者 email、アクション（読みやすいラベルに翻訳）、対象、詳細（展開で JSON）。

**絞り込みとエクスポート**：操作者（email キーワード）、アクション（19 種の特権操作すべてを含むドロップダウン）、開始／終了日（**台北タイムゾーン**の日境界）でリアルタイムに絞り込めます。「CSV エクスポート」は現在の絞り込みで出力（最大 5000 行、UTF-8 BOM 付きで Excel が日中韓文字を正しく開けます）。

**19 種の特権操作**（v7.9.5 以降すべて読みやすいラベルを持ち、バックエンドの書き込み地点と 1 対 1 対応）：MFA リセット、Telegram ID 設定、メンテナンスモード切り替え、連絡先の完全削除、一括完全削除、MCP トークン作成、MCP トークン失効、Email 復活の適用、Hunter 設定変更、Webhook 設定、更新通知の送信、連絡先の統合、DSAR 照会、アカウント状態の設定、ニュースレター送信、GDPR データエクスポート、ロール設定、機能権限の設定、組織設定の変更。本ページは読み取り専用で、バックエンド API も権限を検証します。

## MCP トークンとアクティビティ

パス：`/admin/mcp-tokens`、`/admin/mcp-activity`（いずれも Super Admin のみ）。

**MCP トークン**：外部 AI エージェントが（MCP 経由で）システムにアクセスする際のアクセスキーを管理します。一覧には名前／用途、割り当て先、スコープ、最終使用、有効期限、状態を表示。

- **トークン発行**：名前／説明、割り当て先（エージェントの操作はこのユーザーとして記録）、スコープ（読み取り：contacts/newsletter/tags、書き込み：contacts/notes/newsletter。割り当て先の権限を超えるスコープは警告）、任意アクター許可（任意）、有効期限（無期限／1 年／30 日／24 時間）を入力。発行後、**平文キーは 1 回だけ表示**されるので、すぐにコピー保存してください。
- **管理**：各行でアクティビティへ移動、無効／有効、削除（いずれも確認あり）。無効または期限切れのトークンは使用できません。

**MCP アクティビティ**：エージェントが実行した操作を監査目的で確認します。既定で直近 100 件を読み込み、上部に総数／成功／失敗の統計を表示。ツール（tool）と状態（成功／失敗）で絞り込み可能。列は時刻、ツール、状態、トークン、操作者、IP ハッシュで、「展開」で呼び出しパラメータ（JSON）とエラーメッセージを表示。トークンページのアクティビティリンクから来た場合は、その `token_id` で事前絞り込みされます。

## フィードバック管理

パス：`/admin/feedback`（Super Admin のみ）。

**概要**：ユーザーが送信した提案や不具合報告を確認・トリアージし、ステータスを更新します。タイトル横に総件数を表示し、送信時刻の新しい順に並びます。

**内容と操作**：各カードに種別バッジ（Bug＝赤／機能提案＝紫）、タイトル、送信者、日付、ステータスバッジ（未対応／対応中／対応済み・確認待ち／完了／対応しない）を表示。カードを展開すると全文と添付スクリーンショット（非公開バケットの署名付きリンク）を確認できます。展開したカードの下でステータスボタンを押すと切り替わり、即時保存されます。保存に失敗した場合はエラーを表示します。

**報告者確認制**：「完了」は管理側から直接設定できません。ステータスを「対応済み・確認待ち」にすると、報告者のインターフェース言語で自動的にメール通知が送られます。報告者が解決を確認し `/feedback` で「完了を確認」を押してはじめて「完了」になります。

## 自動化とアラート（スケジュール）

システムは Vercel Cron で複数のバックグラウンドジョブを実行します。これらは（CRON_SECRET／API トークンで）ログインとは独立に自己認証し、実行状況はシステムヘルスの「スケジュール心拍」で確認できます。管理に特に関係するもの：

| ジョブ | 頻度（UTC） | 役割 |
|--------|-------------|------|
| health-watchdog | 10 分ごと | サービス健全性とスケジュール遅延を巡回し、異常時に Telegram で管理者へ通知、復旧時にも通知。同じ問題を繰り返し通知しない |
| check-feedback | 毎日 18:00 | 日次フィードバック要約：新規報告があれば AI の初期所見を添えて管理者へ、なければ「本日は新規 issue なし」を送信 |
| purge-retention | 毎日 19:30 | 保持期間クリーンアップ（下記参照） |
| import-suppressions | 毎日 19:00 | SendGrid 抑制リストを埋め戻し（「配信停止と送信可否」参照） |
| hunter/cron | 毎日 18:00 | email がない連絡先の Hunter.io 自動補完（組織設定の「Hunter.io 自動補完」スイッチで制御） |
| run-report-schedules | 毎時 | 期限到来のレポートスケジュールを自動送信 |
| process-scheduled-campaigns | 10 分ごと | 予約済みニュースレターキャンペーンを処理 |
| task-reminders / pre-meeting-briefings / stale-contacts | 毎日／6 時間ごと／毎週月曜 | タスクリマインダー、会議前ブリーフィング、無連絡連絡先のリマインダー |

**ヘルスウォッチドッグ**：サービスやスケジュールに異常が出ると能動的に通知し、復旧時にも通知します（同じ問題は繰り返しません）。使用量しきい値のアラート（80%／100%）もこの監視に乗り、組織オーナーへ通知されます。

**保持期間クリーンアップ**：期限切れ／論理削除データを毎日削除してテーブルの無限増加を防ぎます——ゴミ箱の連絡先（論理削除から 90 日超）を完全削除、加えて Bot セッション（30 日超）、Telegram 重複排除記録（7 日超）、失効から 30 日超のエージェントトークン、ニュースレター下書きキャッシュ（1 日超）、スケジュール心拍記録（30 日超）。

**Hunter 補完とスイッチ**：Hunter.io は email がない連絡先の email を補完します。毎日のスケジュール実行に加え、システムヘルスページから手動検索もできます。モジュール全体は組織設定からワンクリックで停止でき、停止中はスケジュールも手動検索も credits を消費しません。
