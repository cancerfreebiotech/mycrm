# 聯絡人管理

路徑：`/contacts`

---

## 聯絡人列表

### 搜尋與篩選

- **關鍵字搜尋**：比對姓名、Email、公司名（模糊搜尋）
- **Tag 篩選**：點選 Tag 按鈕只顯示該 Tag 的聯絡人
- **預設排序**：依「最後活動時間」倒序（最近有互動的人在最上面，詳見下方規則）
- **其他排序**：點擊欄位標題切換（姓名 / 公司 / 職稱 / Email / Tags / 建立時間）
- **分頁**：每頁 20 筆，底部頁碼導覽

### 儲存檢視（Saved Views）

把目前套用的搜尋與篩選條件存成一組具名檢視，之後點一下就能重新套用，不必每次重設。檢視為**每位使用者私有**，只有你自己看得到。

列表上方以圓角標籤（chips）顯示你已儲存的檢視（最多 10 個）：

- **套用**：點檢視名稱，立即套用該組條件並更新網址
- **儲存目前檢視**：點虛線的「🔖 儲存檢視」，輸入名稱後即把目前所有篩選（關鍵字、在哪認識、Tag、國家、重要度、語言、Email 狀態、建立者、建立／認識日期範圍等）存成檢視
- **刪除**：點 chip 上的 ✕，二次確認後移除

> 檢視只記錄篩選條件，不含排序與頁碼；套用後會回到第 1 頁。

### 匯出

點擊「Excel」或「CSV」按鈕下載目前篩選結果（不受分頁限制）。

> **需要匯出權限**：僅 **super_admin** 或已獲授予「匯出聯絡人（`export_contacts`）」權限的成員可使用。權限不足時按鈕會停用，滑鼠移上會提示「請聯絡管理員申請匯出權限」。

### 多選與批次編輯

- 每列左側有勾選框，表頭有「全選」（作用於目前分頁）
- 勾選一筆以上後，工具列會出現「**批次編輯（N）**」按鈕，開啟批次編輯視窗
- 可一次套用：認識地點 / 認識日期、介紹人、公司、國家、語言、Tag（Tag 為**附加**，不會移除既有 Tag）
- 欄位留空代表**維持不變**；若填了「認識地點」，系統會同時為每位選取的聯絡人寫入一筆會議互動紀錄

### 寄信給選取／篩選的聯絡人

有選取項目或套用篩選、且名單中有可寄送對象時，工具列會出現綠色「**寄信（N）**」按鈕，點擊後帶著這批收件人跳轉到 `/email/compose` 群發。

- 收件人會**依 Email 去重**（多筆聯絡人共用同一 Email 只寄一次）
- 無法寄送者（沒有 Email、已加入黑名單、已退訂、Email 狀態異常）會自動排除，按鈕旁與上方橫幅會顯示排除數量

### 建立電子報名單

有選取項目或套用篩選、且名單中有可寄送對象時，工具列會出現「**建立名單（N）**」按鈕（需 **super_admin** 或「電子報（`newsletter`）」權限）。

- 點擊後填寫名單名稱（必填）與說明，即可從**可寄送的子集**建立一份電子報名單
- 建立完成後自動跳轉到該名單頁

### 用 LinkedIn 截圖新增聯絡人

工具列的「**新增**」按鈕為下拉選單，包含兩個選項：
- **新增聯絡人**：前往 `/contacts/new` 手動填寫
- **LinkedIn 截圖**：選一張 LinkedIn 個人頁截圖，AI 解析後預先填入新增聯絡人表單

## 新增聯絡人

路徑：`/contacts/new`

分為四個區塊手動填寫：

| 區塊 | 欄位 |
|------|------|
| **基本資訊** | 姓名（中/英/當地語言）、公司（中/英/當地語言）、職稱 |
| **聯絡方式** | Email、第二 Email、電話、第二電話、地址、網站、**國家** |
| **社群媒體** | LinkedIn、Facebook |
| **備註** | 備註文字 |

### 名片 AI 辨識（多張）

1. 點擊「選擇名片照片」，一次選取最多 **6 張**（正面、反面等）
2. 點擊「掃描名片」，系統壓縮後合併送 AI 辨識
3. 頁面左側顯示照片預覽，右側顯示 AI 辨識結果對照
4. 點擊「套用到表單」填入欄位，或「忽略」手動填寫

> 存檔前系統會比對 Email 和姓名相似度，偵測到重複聯絡人時會提示確認。

## 批次上傳

路徑：`/contacts/batch-upload`

一次上傳最多 **50 張**名片照片：

1. 點擊或拖曳上傳圖片（支援 JPG / PNG）
2. 系統同時處理最多 5 張（並行上傳 + AI 辨識）
3. 進度條顯示目前進度
4. 辨識完成後顯示預覽表格，可逐筆修改
5. 批次重複偵測（與現有聯絡人比對）
6. 點擊「全部存檔」一次儲存

## 待審核名片

路徑：`/contacts/pending`

透過 Telegram Bot **批次模式（`/b`）** 上傳的名片，AI 辨識完成後會先進到這裡等待確認，不會直接變成聯絡人。頁面每 5 秒自動刷新仍在辨識中的項目。

### 篩選

- **狀態**：全部 / 辨識中 / 已辨識 / 失敗
- **上傳者**：當有多位上傳者時，可依上傳者過濾

### 每張名片可操作

**辨識中**：顯示辨識進度，等待完成。

**已辨識**：可先就地編輯欄位（姓名、公司、職稱、Email、電話、國家）、重要度（H / M / L）、語言、Tag，以及「認識地點 / 日期（Met at）」，接著選擇：

| 操作 | 說明 |
|------|------|
| **確認存檔** | 存成新聯絡人 |
| **合併到 X** | 系統偵測到疑似重複時，合併到既有聯絡人（可選「補空欄」或「取代」） |
| **合併到既有…** | 手動搜尋並選擇要合併的聯絡人 |
| **刪除** | 刪除此待審核名片 |
| **看原圖** | 開啟名片原圖 |

> 已辨識的名片可用上方「批次設定 Met at」，一次把相同的認識地點 / 日期套用到所有已辨識項目。

**失敗**：顯示錯誤原因，並提供「**重試辨識**」重新送 AI 辨識。

### 重跑卡住的辨識

若有屬於你、長時間卡在「辨識中」的名片，頁面右上會出現「**重跑卡住的辨識**」按鈕，點擊即可把卡住的項目重新送回辨識佇列。

## 辨識失敗名片

路徑：`/contacts/failed-scans`

透過 Bot 上傳、但 AI **完全無法辨識（連姓名都讀不到）** 的名片圖片會落在這裡。此頁只顯示**你自己**上傳的失敗圖片。

- 以縮圖網格顯示所有失敗圖片與上傳時間
- 點擊圖片或「看原圖」可放大檢視
- 可單張刪除，或勾選（含「全選」）後以「刪除已選」批次刪除

> 要重新辨識，請用 Telegram Bot 重拍或重新上傳該名片。

## 聯絡人詳情

路徑：`/contacts/[id]`

### 基本資料區

顯示所有聯絡人欄位，右上角「編輯」按鈕開啟編輯 Modal。

- **Email / 電話**欄位旁有複製圖示，點擊即複製到剪貼簿
- **國家**欄位顯示旗幟 emoji + 中文名稱（如 🇹🇼 台灣）

### 名片管理

一個聯絡人可以掛多張名片（正面、反面、或自訂標籤）：
- 一次選取最多 **6 張**，AI 合併辨識後顯示差異欄位供確認
- 點擊卡片右上角 🗑 刪除

### 互動紀錄時間軸

依時間倒序顯示所有紀錄，類型包含：
- **筆記** (note)
- **會議** (meeting)
- **Email** (email)
- **系統** (system) — 系統自動產生

預設載入 20 筆，捲動到底自動載入更多（無限捲動）。

### 發送 Email

點擊「寄信」按鈕，開啟增強版寄信 Modal：
- **收件人（To / CC / BCC）**：可手動修改，預設帶入聯絡人 Email
- **範本**：選擇後主旨、內文、附件自動帶入
- **AI 生成**：輸入描述，AI 帶入收件人資訊與最近互動紀錄自動產生內文（HTML 格式，TipTap 編輯器）
- **聯絡人照片**：若該聯絡人已透過 bot `/p` 上傳過合照，modal 會顯示縮圖區塊，點一下縮圖即可附加到信件，再點一下移除
- **臨時附件**：上傳本次專用附件（單檔 5MB），不儲存到範本

發送透過 Microsoft Graph（Outlook）送出，自動建立一筆 Email 互動紀錄並將聯絡人推到「最後活動」排序頂端（見下一節）。

### 會議前 Briefing

詳情頁的「會議前 Briefing」區塊，可請 AI 整理這位聯絡人與其公司的最新公開動態，供拜訪或開會前參考：

- 點擊「產生 Briefing」後，系統會在背景**非同步**處理（通常一分鐘內完成，期間可離開頁面）
- 完成後結果直接顯示在同一區塊，並附上可查證的資料來源連結
- 產生過後，按鈕會變為「重新產生」

> 詳見 [會議前 Briefing](social-briefing.md)。此摘要**不會**寫入互動紀錄。

## 最後活動時間 (`last_activity_at`)

**目的**：讓最近有互動的聯絡人自動浮到列表頂端，不用記名字去搜尋。

### 計算規則
`contacts.last_activity_at` 由 DB trigger 自動維護，取下列時間戳的**最晚者**：
1. `interaction_logs.created_at` — 限 type 為 `note` / `meeting` / `email` 且**非 SendGrid 寄送**
2. `contact_photos.created_at` — 任何照片上傳（bot `/p` 或網頁上傳）
3. 若以上都沒有 → fallback 到 `contacts.created_at`

### 行為矩陣

| 動作 | 算最後活動？ | 寫互動紀錄？ |
|---|---|---|
| 聯絡人詳情頁寄信（Outlook/Graph） | ✅ | ✅ |
| Bot `/met` 拜訪筆記 | ✅ | ✅ (type=note 或 meeting) |
| Bot `/meet` 會議 | ✅ | ✅ (type=meeting) |
| Bot `/work` 任務 | ✅ | ✅ (type=note) |
| Bot `/p` 合照上傳 | ✅ | ✅（附註則寫系統紀錄） |
| `/email/compose` Outlook 群發 | ✅ | ✅ |
| **`/email/compose` SendGrid 群發** | ❌ | ✅ |
| **Newsletter campaign（一律走 SendGrid）** | ❌ | ✅ |
| Bot `/a` 新增名片（AI 辨識） | ❌ (type=system) | ✅ |

### 為什麼 SendGrid 不算？
SendGrid 典型用途是大量自動化群發（newsletter、行銷信）。如果每封 SendGrid 信件都刷新 `last_activity_at`，寄一次 newsletter 給 4000+ 訂閱者會把每個聯絡人的「最後活動」洗成同一時間，這個排序訊號就完全失去意義。Outlook / Graph 寄信是 1-to-1 或小範圍互動，才是真正值得追蹤的「我最近跟誰聊過」訊號。

**使用者仍可從互動紀錄時間軸看到完整寄送歷史**，只是不影響排序。

### UI 提示
`/email/compose` 切換到 SendGrid 時會顯示紫色 badge：
> ⚠ SendGrid 寄送不會計入聯絡人最後活動時間（仍會寫互動紀錄）

## 冷關係提醒（每週 Telegram 推播）

系統每週一（UTC 02:00）自動檢查你名下的聯絡人，把「太久沒互動」的人整理成一則清單，透過 Telegram 主動提醒你「該聯繫了」。只有**已綁定 Telegram** 的成員會收到。

### 判斷規則

- **對象**：`created_by` 為你的聯絡人（你自己建立的），不含已刪除者
- **變冷門檻**：以「最後活動時間」（`last_activity_at`，無紀錄則退回建立時間）計算——**高重要度超過 30 天**、其餘超過 **90 天** 未互動，即視為冷關係
- **已在跟進者不重複提醒**：若該聯絡人已有進行中（`pending`）的任務，代表已有人在處理，會自動排除
- **每則最多 8 位**：依冷卻天數由久到近排序，只列最該處理的前 8 位
- **沒有冷關係就不發**：當週沒有符合條件的聯絡人時，不會收到任何訊息（不打擾）

### 提醒內容

清單每筆顯示姓名（公司）與「N 天未互動」，並附上該聯絡人頁面連結，最後提示可直接回覆 `/v <名字> <重點>` 用一句話記錄互動（詳見 [Bot 指令列表](../bot/commands.md)）。

```
🌿 該聯繫了

• 王大華（ABC 公司）— 45 天未互動
  https://…/contacts/…

回覆 /v <名字> <重點> 可一句話記錄互動
```

## Hunter.io Email 自動查詢

**目的**：新建聯絡人若名片/LinkedIn/表單沒有 email，系統背景向 Hunter.io Email Finder 查一次。查到就寫進聯絡人並通知（bot）/ 背景更新（web）。

### 自動查的觸發點
- Bot `/a` 名片辨識完成、結果沒 email
- Bot `/li` LinkedIn 辨識完成、結果沒 email
- Bot `/p 姓名` 或 `/p 姓名 | 公司` 找不到 → 按「✅ 建立新聯絡人」→ 查
- 網頁批次上傳每筆 insert 後、該筆沒 email
- 網頁手動新增 `/contacts/new` submit 後、email 欄空
- **網頁編輯聯絡人**時，email 仍空且補上了 `company` 或 `name_en` → 自動 reset `hunter_searched_at` 並重查

### Bot 回報
Bot 每次觸發 Hunter 都會回訊息說明結果（避免你以為沒查）：
- 📧 `Hunter 已自動查到 email：X`
- 🔍 `Hunter 查過了，目前沒找到 email（cron 30 天後會再試一次）`
- ℹ 資料不足未查 — 說明「中文名需補英文名」或「需補公司」才能查，補完 cron 會重試
- ⚠ API 錯誤 — 30 天後自動重試

### Hunter 的硬限制
- **Hunter 不接受純中文/日文/韓文名**：必須有英文拼音的 first_name + last_name
- **必須提供公司名**：Hunter Email Finder 靠公司推測 email 格式
- 所以中文聯絡人建議補上 `name_en` 欄位（例如「戴建丞」→ 補「Chien-Cheng Tai」）+ 公司 = 下次 cron 會重查

### 每日自動批次（cron）
- 每天 **02:00 Asia/Taipei** 跑一次 `/api/hunter/cron`
- 每次處理最多 50 筆「沒 email 且 30 天內沒查過」的聯絡人
- 800 筆舊帳約 16 天清一輪
- Hunter 帳號剩餘 credits 低於 5 時會自動 skip 該次執行

### 額度規則
Hunter Free tier = 50 searches/month，但「**找不到不扣 credit**」。所以大量舊聯絡人（多半找不到）可以積極查不怕爆表。管理頁 `/admin/hunter` 顯示實時剩餘額度。

### 管理頁
`/admin/hunter` 仍保留：
- 統計（總沒 email / 從未查過 / 查過沒結果 / 本月查過）
- 手動觸發批次（最多 100 筆/次）
- 設定 API key
- 重置 `hunter_searched_at`（需要重新掃一輪時用）

| 欄位 | 說明 |
