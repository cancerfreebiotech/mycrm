# 系統健康

路徑：`/admin/health`（僅 `super_admin`；側邊欄僅對 super_admin 顯示，資料 API 亦於後端驗證權限）

---

## 這頁在做什麼

一頁綜覽整個系統的運作狀態，包含外部服務健康、Email 搜尋工具、以及自 v7.4.0 加入的**排程心跳、死信、本月用量**三個區塊。

頁面頂端可「立即檢查」，或勾選「每 30 秒自動重整」持續監控；並顯示上次檢查時間。

---

## 外部服務狀態

摘要列統計正常／異常／未設定的服務數，下方以卡片列出各服務：Supabase、Gemini、Telegram Bot、SendGrid、Teams Bot。

- 狀態徽章：**正常** / **異常** / **未設定**。
- 每張卡片附延遲條：綠色（< 500 ms）、黃色（< 2000 ms）、紅色（更慢），底部有色標說明。

---

## Hunter.io（Email 搜尋）

針對「沒有 email」的聯絡人，用 Hunter.io 嘗試找出對方 email。

- **API Key**：貼上金鑰後儲存；已設定會顯示綠色勾。
- **統計**：無 email 總數、未搜尋過、搜尋未果、本月已搜尋；若有額度資訊會顯示本月剩餘 credits。
- **開始搜尋**：對尚未搜尋的聯絡人發動查詢，結束後列出找到／未找到清單。需先設定 API Key。
- **重設搜尋狀態**：清除搜尋標記，讓聯絡人可再次被搜尋（需二次確認）。
- **模組停用提示（v7.9.6）**：若「Hunter.io 自動補全」已在[組織設定](org-settings.md)關閉，執行搜尋會顯示「模組已停用」提示與前往組織設定的連結，且不會實際查詢或消耗額度。

---

## 排程心跳（v7.4.0）

列出各背景排程工作（cron）的最近執行狀況：

- 狀態：**正常** / **逾時** / **錯誤** / **無紀錄**；有逾時或錯誤時整區會以警示色標示。
- 顯示上次完成時間（相對時間）與執行耗時。

用來快速確認夜間自動化、回報檢查等排程是否正常在跑。

---

## 死信（v7.4.0）

列出各資料表中處理失敗的項目數（dead letters）：

- 有失敗筆數的資料表會以紅色標示，可展開查看最近的錯誤訊息。
- `pending_contacts`、`contact_briefings` 提供「重新排入」按鈕，把失敗項目重新丟回佇列。
- `failed_scans` 提供前往「辨識失敗審查」的連結。

---

## 本月用量與預算門檻（v7.4.0；門檻編輯 v7.9.5）

以卡片顯示本月各項用量並對照上月：AI 呼叫次數、AI 輸入／輸出 tokens（若有紀錄）、寄送 Email 數、電子報寄送數。

自 v7.9.5 起，每張卡片可直接填「月上限」門檻，按「儲存門檻」保存：

- 留空＝**不設限**，卡片顯示與上月的對比。
- 設定上限後，卡片改顯示用量進度條與百分比：**綠 < 80%**、**黃 80–99%**、**紅 ≥ 100%**。
- 達 80% / 100% 時，系統會透過既有監控自動通知組織擁有者（告警本身為既有功能）。

---

## 注意事項

- 本頁僅 super_admin 可用；各區塊分別向對應 API 取數，某區失敗只會該區顯示錯誤並可單獨重試，不影響其他區塊。
- Hunter.io 免費額度有限，搜尋前請留意剩餘 credits。
