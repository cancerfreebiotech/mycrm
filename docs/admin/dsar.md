# 資料主體查詢 (DSAR)

路徑：`/admin/dsar`（僅 `super_admin`）

---

## 這頁在做什麼

輸入一個 email，找出系統中所有與該地址相關的聯絡人，並列出每一筆的**個資足跡**（互動紀錄、名片、電子報收件數）。用來回應當事人（資料主體）的資料查詢／匯出／刪除請求。

- 依 email **精確比對**（不分大小寫）聯絡人的 `email` 或 `第二 email`。
- **包含已刪除（回收區）的聯絡人**——資料主體請求必須涵蓋所有紀錄，不論是否已被軟刪除；已刪除者會標上「已刪除」標記。
- 同時顯示該地址的**寄送資格判定**（可否寄信及原因），判定邏輯與[退訂與寄送資格](suppressions.md)共用。
- 每一筆聯絡人可直接「檢視」或「匯出」個資。

---

## GDPR 情境

DSAR（Data Subject Access Request，資料主體查詢請求）是 GDPR 賦予當事人的權利：當事人可要求得知、取得或刪除公司持有的個人資料。這頁讓 super_admin 在收到請求時，能：

1. 以 email 為索引，找出**所有**與該人相關的紀錄（含回收區）——對應「查閱權」。
2. 量化資料足跡，並逐筆匯出完整個資 JSON——對應「查閱權／資料可攜權」。
3. 確認該地址目前的寄送／抑制狀態——對應「反對權／刪除後續」。

> 為了可歸責性，**每一次查詢都會寫入稽核日誌**（動作：`dsar_lookup`，即[稽核日誌](audit-log.md)中的「DSAR 資料查詢」），對象記為查詢的 email。

---

## 誰能用

- 僅 `super_admin` 可開啟本頁。
- 後端 API `/api/admin/dsar` 會**再次驗證權限**：未登入回 401、非 super_admin 回 403（`/api/admin/*` 不經 auth middleware，故 route 自行把關）。

---

## 怎麼用

1. 在搜尋框輸入**完整** email（格式錯誤會提示「請輸入有效的 email」）。
2. 按「查詢」。
3. 上方出現該地址的寄送資格卡片；下方列出所有相符的聯絡人與其個資足跡。
4. 每列可：
   - **檢視** → 開啟該聯絡人頁面 `/contacts/{id}`。
   - **匯出** → 下載該聯絡人的完整個資 JSON（見下）。

找不到相符聯絡人時顯示「找不到符合的聯絡人」；尚未查詢時顯示輸入提示。

---

## 結果欄位說明

| 欄位 | 說明 |
|------|------|
| 姓名 | 聯絡人姓名；已軟刪除者附「已刪除」標記 |
| 公司 | 聯絡人公司 |
| 建立者 | 建立此聯絡人的使用者（顯示名稱） |
| 建立時間 | 聯絡人建立日期 |
| 互動紀錄 | 該聯絡人的 `interaction_logs` 筆數 |
| 名片 | 該聯絡人的 `contact_cards` 筆數 |
| 電子報收件 | 該聯絡人的 `newsletter_recipients` 筆數 |

---

## 個資匯出（GDPR 資料匯出）

每列的「匯出」呼叫 `/api/contacts/{id}/export`，回傳單一 JSON 檔（`contact-{id}-export.json`），彙整該聯絡人的所有個資：

- 聯絡人主檔、名片、相片、互動紀錄、任務、電子報收件、Email 事件。
- 名片與相片的**影像本體存於 Storage**，匯出檔只含網址／路徑，不含影像位元組。

> 匯出權限為 super_admin 或具備「匯出聯絡人」功能權限；super_admin 兩者皆符合。匯出動作同樣會寫入[稽核日誌](audit-log.md)（動作：GDPR 資料匯出），並記錄各類資料筆數。

---

## 注意事項

- 本頁僅 super_admin 可用，後端 API 也會再次驗證權限。
- 這裡是唯讀查詢頁——不會在此修改或刪除任何資料；刪除請於[回收區](trash.md)或聯絡人頁操作。
- 寄送資格的完整來源與判定說明，見[退訂與寄送資格](suppressions.md)。
