# 配信停止・同意管理

パス：`/admin/suppressions`（`super_admin` のみ）

---

## このページの目的

任意のメールについて「**送信できるか、できない理由は何か**」を確認し、システム全体の抑制（サプレッション）状態を俯瞰します。判定は**厳密な和集合**です：いずれかのソースが抑制していれば、送信不可と判定されます。

このページには 2 つのビューがあります：

- **単一メールの照会**：アドレスを入力 → 送信可否の判定カードと、5 つのソースごとの状態テーブルを表示。
- **未入力（デフォルト）**：ソース別の**集計数**と、**直近 50 件**の抑制記録（ソース横断で新しい順）を表示。

> このページは閲覧専用です——抑制状態を反映しますが、ここで抑制の追加や解除は行いません。

---

## 5 つの抑制ソース

「送信できるか」は、システム各所に分散した次の 5 つのソースで決まります（ソースコードに準拠。判定の意味はニュースレター送信 worker `src/lib/newsletter-send-worker.ts` と CRM 直接送信パス `/api/email/send` に整合）：

| ソース | 条件 | 説明 |
|--------|------|------|
| 連絡先オプトアウト | `contacts.email_opt_out = true` | CRM 直接送信パスがこのフラグを尊重 |
| 連絡先メールステータス | `contacts.email_status` が**非 null** | 送信 worker は**あらゆる**非 null ステータスで抑制。値は `bounced`／`invalid`／`unsubscribed`／`recipient_blocked`／`spam_report` など |
| ブラックリスト | `newsletter_blacklist` に存在 | 通常は SendGrid のハードバウンス／無効メール／ブロック由来 |
| 全体配信停止 | `newsletter_unsubscribes` に存在 | 全体ブロック。SendGrid の配信停止・スパム報告を含む |
| 購読者の配信停止 | `newsletter_subscribers.unsubscribed_at` が**非 null** | ニュースレター購読者本人による配信停止 |

照合ルール：

- 「連絡先オプトアウト」「連絡先メールステータス」は連絡先の `email` または `第二 email` を照合し、**未削除**（`deleted_at IS NULL`）の連絡先のみを対象とします。
- ブラックリスト／全体配信停止／購読者の配信停止は email 列を完全一致（大文字小文字を区別しない）で照合します。

---

## 毎日の SendGrid 抑制インポート（cron）との関係

このページが読むソースのいくつかは、スケジュールジョブ `POST/GET /api/sendgrid/import-suppressions` が毎日**バックフィル**しています：

- スケジュール：`vercel.json` の cron `0 19 * * *`（UTC 19:00、台北ではおおよそ翌日 03:00）。
- 内容：SendGrid の直近 90 日の抑制リスト（5 種類）をページ送りで取得し、本システムに書き戻します：

| SendGrid リスト | `contacts.email_status` を設定 | 追加で書き込み | 対話記録 |
|-----------------|------------------------------|----------------|----------|
| ハードバウンス | `bounced` | 非 CRM アドレス → ブラックリスト | 「SendGrid 硬退信」 |
| 無効メール | `invalid` | 非 CRM アドレス → ブラックリスト | 「SendGrid 無效信箱」 |
| 配信停止 | `unsubscribed` | 全体配信停止（ソース `sendgrid_import`） | 「SendGrid 已退訂」 |
| ブロック | `recipient_blocked` | 非 CRM アドレス → ブラックリスト | 「SendGrid 被擋下」 |
| スパム報告 | `spam_report` | 全体配信停止（ソース `sendgrid_spam_report`） | 「SendGrid 垃圾信檢舉」 |

- すでに CRM にある連絡先は `email_status` を更新するのみ（ブラックリストへは重複追加しない）。ブラックリストへ入るのは非 CRM アドレスだけです。
- 一致した CRM 連絡先には system 対話記録を 1 件追加します（同種は重複作成しません）。
- super_admin が手動でも実行できます（ニュースレターのリストページのインポートボタン）。cron の実行はハートビートを記録し、[システムヘルス](health.md)ページで確認できます。

> 一言でいえば：**このページはビューアにすぎません**。SendGrid のバウンス／配信停止／苦情を「連絡先メールステータス／ブラックリスト／全体配信停止」に実際に流し込んでいるのは、この毎日の cron です。

---

## 利用できる人

- 本ページを開けるのは `super_admin` のみです。
- バックエンド API `/api/admin/suppressions` は権限を再検証します：未ログインは 401、super_admin でなければ 403。

---

## 使い方

1. **単一アドレスの確認**：完全なメールを入力（形式が不正なら警告）→「確認」を押す →判定カードとソース別状態テーブルを確認。
2. **全体状態の確認**：入力欄を空のままにするとデフォルトビュー → ソース別集計 + 直近 50 件の抑制記録。

---

## 列の説明

### ソース別状態テーブル（単一アドレス照会時）

| 列 | 説明 |
|------|------|
| ソース | 上記 5 つのソースのいずれか |
| ステータス | 「抑制中」（赤）または「正常」（緑） |
| 詳細 | そのソースの補足情報（ステータス値、配信停止理由、配信停止日時など） |

### 最近の抑制記録（デフォルトビュー）

| 列 | 説明 |
|------|------|
| メール | 抑制されたアドレス |
| ソース | 抑制ソース |
| 詳細 | 補足情報 |
| 日時 | その抑制が発生した日時（新しい順） |

---

## 注意事項

- 本ページは super_admin のみ利用でき、バックエンド API でも権限を再検証します。
- 閲覧専用のビューです——抑制を解除するには、そのソース側で対応してください（連絡先のメールステータス、[Email 復活](email-recovery.md)、配信停止リストの保守など）。
- 同じ判定ロジックは、ニュースレター／一括送信フローの受信者除外にも使われます。
